# `/stock-guide` — sub-PRD

Owner: `worker_dash-stock-guide`. Reports to `worker_subgerente-app`.

> **Visual identity note:** Stock Guide is **NOT** the Market Watch / trading-terminal module. It uses the **standard dashboard identity** (brand orange `#ff5000`, Arial, liquid-glass cards) exactly like `/subsidy-tracker` and the other Fase-3 dashboards — never the scoped `.stocks-dark` / `.stocks-light` theme owned by `/stocks`. The only thing it borrows from Market Watch is the Yahoo quote proxy (`/api/stocks/quote`) and the `useStockQuote` hook.

## Overview

Equities-research module for the Brazilian Oil & Gas + Fuel Distribution coverage universe. Two analyses, one shared brain:

1. **Comps table** — one row per covered company with the headline research numbers: target price, recommendation (OP/MP/UP), and six forward column pairs (EV/EBITDA, P/E, FCFE Yield, Div Yield, EBITDA, Volumes), each split into two forward years (`config.y1_label` / `config.y2_label`, e.g. 2026E / 2027E). **Market cap**, **upside**, and the **four price-sensitive multiples (EV/EBITDA, P/E, FCFE Yield, Div Yield)** are computed LIVE (never stored) — see below. Only EBITDA and Volumes are direct data.
2. **Per-company sensitivity** — a freeform 2D matrix (admin-defined axes/labels/cells) for the selected company, opened below the table (desktop) or in a BottomSheet (mobile).

Targets, shares, the **fundamentals** (net debt, EBITDA, net income, FCFE, dividends), volumes, sensitivities and config are **admin-only inputs** maintained in the Admin Panel → Stock Guide section. The four price-sensitive multiples are **NOT stored** — they are always derived live from the price + fundamentals (no admin override). Any company can be **hidden** (regulatory restriction); hidden companies are removed from the table and listed only by name in a "Currently restricted" footnote — their financials never reach a non-admin browser.

Audience: **Client + Admin**, hidden from anonymous public (`is_visible_for_public=false`). One-click reconfigurable in Admin Panel → Permissions.

Slug / URL: `stock-guide` → `/stock-guide`. NavBar placement: **standalone top-level NavBar entry** ("Stock Guide", alongside Market Watch / News Hunter), not under any dropdown.

## Live derivations (frontend, in the hook)

Computed per visible row from the batched Yahoo quote + the admin-input fundamentals — **not** stored server-side. **All monetary inputs are BRL million**, so EV/EBITDA and P/E are dimensionless and the yields are ×100 for percent points.

| Field | Formula | Null-safety |
|---|---|---|
| `livePrice` | `quote.regularMarketPrice` matched on `yahoo_symbol` (fallback `ticker`) | `null` if the quote is missing |
| `marketCapBrlMn` | `shares_outstanding × livePrice / 1e6` (BRL mn) | `null` if either input is `null` |
| `upsidePct` | `target_price / livePrice − 1` | `null` unless `livePrice > 0` and `target_price` present |
| `evBrlMnY1` / `evBrlMnY2` | `marketCapBrlMn + net_debt_yN` (BRL mn). Net debt is **forward, per year**; either may be negative (net cash → lowers EV). Market cap stays a single current value. | `null` if either input is `null` |
| `evEbitdaY1` / `evEbitdaY2` | `evBrlMnYN / ebitda_yN` | `null` unless `ebitda_yN > 0` (EBITDA ≤ 0 → not meaningful → `—`) |
| `peY1` / `peY2` | `marketCapBrlMn / (net_income_adj_yN ?? net_income_yN)` — uses the ADJUSTED net income when present (e.g. Vibra, which strips non-recurring tax credits), else falls back to the reported net income | `null` unless the chosen earnings `> 0` (non-positive earnings → P/E not meaningful → `—`) |
| `fcfeYieldY1` / `fcfeYieldY2` | `(fcfe_yN / marketCapBrlMn) × 100` percent | `null` unless `marketCapBrlMn > 0`; FCFE may be negative → negative yield shown |
| `divYieldY1` / `divYieldY2` | `(dividends_yN / marketCapBrlMn) × 100` percent | `null` unless `marketCapBrlMn > 0` |

Everything renders `—` (never `NaN`) when null. Every divide-by-zero / non-positive denominator is guarded. Market cap and EV land in BRL because `shares_outstanding` is the absolute share count and the live price is in BRL; net debt and the per-year fundamentals are all BRL mn. **EV convention:** `EV(year) = Market cap + Net Debt(year)` — net debt is now **forward per year** (`net_debt_y1` / `net_debt_y2`), so EV is computed per forward year. Market cap remains a single current value. **No override:** the four multiples are *always* computed — there is no admin-stored fallback.

The 4 live multiples render `—` while `quotesLoading` (they depend on the live price); EBITDA and Volumes are direct data and never gate on the quote.

### Live-quote wiring & cadence

- The hook collects the `yahoo_symbol` (fallback `ticker`) of **visible** rows into one de-duplicated list and passes it to `useStockQuote(symbols)` → a single batched `GET /api/stocks/quote?tickers=…` request (the proxy auto-appends `.SA` for B3 tickers and returns an array).
- Cadence: **one-shot fetch on load + a manual "Refresh quotes" button** (desktop only). **No polling ticker** — comps are snapshots and the Yahoo proxy is per-IP rate-limited.
- Hidden companies have a `null` `yahoo_symbol` AND are excluded from `visibleRows`, so a restricted ticker is structurally absent from the quote request — the browser cannot fetch a restricted name's price even if it tried.

## Hide-aware RLS posture

The three tables have **RLS enabled with NO SELECT policy and NO write policy** (deliberately stricter than `field_stakes`). With RLS on + zero policies, direct PostgREST returns **0 rows** for non-superusers — `supabase.from('stock_guide_companies').select('*')` yields `[]` for anon/client. All access flows through SECURITY DEFINER RPCs, which is what keeps hidden financials server-side.

`get_stock_guide_comps()` is the hide-aware core: it returns one row per company in `display_order`, but for hidden rows seen by a non-admin it forces **every** field except `ticker` / `company_name` / `is_visible` / `display_order` to NULL (including `sector`, `volume_unit`, `yahoo_symbol`, `shares_outstanding` and all comps). Admins (`is_admin()`) receive every field through the same call. `get_stock_guide_sensitivity(ticker)` returns `{}` for a hidden company unless the caller is an admin.

### Dynamic drivers (live market-data catalog)

Source of truth: `src/hooks/useMarketDrivers.ts` (frontend-only — no DB compute). DB `source` column was added by `worker_supabase` (commit `8c3c084f`, migration `20260607000000`).

A driver in `stock_guide_drivers` is now either:

- **Static** — `source` is null/`''`; the admin types `current_value`; the effective "today" value is `current_value`.
- **Dynamic** — `source` is a key in the **market-driver catalog** below; `current_value` is ignored (may be null) and the effective value is **computed LIVE in the browser** from the Yahoo proxy. The unit is auto-filled from the catalog.

**Catalog** (`MARKET_DRIVER_CATALOG`, 4 metrics):

| `source` key | Label | Unit | Computation |
|---|---|---|---|
| `avg_brent_2026` | Avg Brent 2026 | USD/bbl | month-average of 12 (realized + forward, see below) |
| `avg_brent_2027` | Avg Brent 2027 | USD/bbl | month-average of 12 (all forward, fallback spot) |
| `avg_fx_2026` | Avg FX (USD/BRL) 2026 | BRL/USD | month-average of 12 (realized + **spot-flat** approx) |
| `avg_fx_2027` | Avg FX (USD/BRL) 2027 | BRL/USD | spot, held flat (= spot) |

**Inputs** (3 GETs to the existing proxy, **once on mount**, no polling — these are slow-moving macro assumptions and the proxy is per-IP rate-limited):

- `GET /api/stocks/futures-curve` → `contracts[{ month /*0-indexed*/, year, price }]` — monthly Brent forward curve (~24 contracts, starts ~M+2, server-cached 24h).
- `GET /api/stocks/history?ticker=BZ=F&range=1y&interval=1d` → Brent realized daily history `[{ date /*unix sec*/, close }]`.
- `GET /api/stocks/history?ticker=USDBRL=X&range=1y&interval=1d` → USD/BRL realized daily history.
- `GET /api/stocks/quote?tickers=BZ=F,USDBRL=X` → spot fallbacks (`regularMarketPrice`; fallback to last realized close).

**Computation** (uses the REAL current date `new Date()` in the browser; `now=new Date()`, `curYear=now.getFullYear()`, `curMonth=now.getMonth()+1` ∈ 1–12). Helpers: `realizedMonthlyAvg(history, year, month)` = mean of `close` over candles whose **UTC** date is in {year, month} (null if none); `forwardPrice(curve, year, month)` = price of the contract with `year === c.year && month === c.month+1` (null if absent / non-positive).

- **`avg_brent_<Y>`** — mean of the 12 monthly values, each by month `m=1..12`:
  - past (`Y<curYear` OR (`Y===curYear` AND `m<curMonth`)) → `realizedMonthlyAvg(brentHist, Y, m)`;
  - current (`Y===curYear` AND `m===curMonth`) → `realizedMonthlyAvg(brentHist, Y, m) ?? brentSpot` (month-to-date, falls back to spot);
  - future → `forwardPrice(curve, Y, m) ?? brentSpot` (the near 1–2 months may not be in the curve since it starts ~M+2 → fall back to spot).
  - Result = mean of the non-null monthly values. For Y=2027 every month is future.
- **`avg_fx_<Y>`** (spot-flat approximation — there is no FX forward in the proxy): past → `realizedMonthlyAvg(fxHist, Y, m)`; current + future → `fxSpot` held flat. For Y=2027 every month → `fxSpot`, so the mean = `fxSpot`.

**Null-safe throughout:** if the inputs are missing, the metric value is `null` and the UI renders "—".

**Wiring.** `useMarketDrivers()` returns `{ values: Record<key, number|null>, loading, error, catalog }` plus two exported helpers used by both consumers:

- `resolveDriverValue(driver, marketValues)` → the driver's effective today value (`marketValues[source]` when `source` is a catalog key, else `driver.current_value`).
- `isDynamicSource(source)` → true when `source` references a known catalog key.

The hook is reused by **both** the dashboard brain (`useStockGuideData`) and the admin-panel Drivers editor, so the same numbers drive the sensitivity-table highlight AND the admin "Computed: …" preview. In `useStockGuideData`, `resolveDriverAxis(axis)` now also returns a resolved `currentValue` (via `resolveDriverValue`); both Views highlight / interpolate / caption on this `currentValue` instead of the raw `driver.current_value`, so a dynamic driver tracks the live market value. `marketValues` + `marketDriversLoading` are exposed on the hook for raw-map consumers.

**Admin Drivers editor** (`admin-panel`, desktop-only): each driver row has a **Source** select — **Static** (default, `source=''`) + the 4 catalog metrics by label. Picking a catalog metric sets `source`, auto-fills `unit` from the catalog (unit input disabled), disables the `current_value` input, and shows the live "Computed: {value} {unit}" (FX metrics carry a "(spot-based approximation — no FX forward)" note). A "dynamic · {value}" badge renders under the name. On save, a dynamic driver sends `source` + catalog `unit` + `name` + `display_order` with `current_value: null`; a static driver is unchanged (`source: ''`). The `rpcAdminUpsertStockGuideDriver` wrapper is generic — it forwards `source` untouched.

### Hook (`useStockGuideData.ts`)

- Fetches drivers + sensitivity tables in the same batched `Promise.all` as comps + config. Exposes `drivers: StockGuideDriver[]`, `sensitivityTables: SensitivityTable[]`, and derived `selectedTables` = tables where `selectedTicker ∈ table.companies` (sorted by `display_order`). `selectedTicker`/`selectTicker` kept (default = first visible company); the per-table fetch is gone (tables arrive in the initial batch).
- `computeSensitivityCell(table, rowIdx, colIdx) → { value, unit }` — the live-derived cell helper. Resolves the cell's company (row company axis → `companies[rowIdx]`; else col company axis → `companies[colIdx]`; else single-company `table.companies[0]`), looks up that company's live `livePrice` + `marketCapBrlMn` from a `liveByTicker` index (built from `visibleRows`, NOT the sector-filtered `computedRows`, so a cell always resolves even when the sector filter hid that company), then applies `value_mode`: `absolute`→primary; `yield`→`primary/marketCap*100` (guard marketCap>0); `pe`→`marketCap/primary` (guard primary>0); `ev_ebitda`→`(marketCap+secondary)/primary` (guard primary>0, secondary≠null, marketCap≠null); `upside`→`(primary/livePrice−1)*100` (guard livePrice>0). Null-safe → `null` → render "—". The returned `unit` is the table's own unit for `absolute`, `'%'` for yield/upside, `'×'` for pe/ev_ebitda. `formatSensitivityCell(value, unit)` (exported) formats it.
- `resolveDriverAxis(axis) → { driver, scenarios }` — looks up `driver_id` in `drivers`.

### UI (both views, dual-view binding honored)

Below the comps table, the selected company's `selectedTables` render (desktop: panel below; mobile: in the BottomSheet). Each table shows its `title` + a `metric · unit` badge, then the 2D matrix. Axis headers: company axis → company names (the `selectedTicker`'s row/col highlighted orange); driver axis → scenario + driver unit, with a "Current: {driver.name} = {current_value} {unit}" caption and the row/col matching `current_value` highlighted — if the current value falls strictly between two scenarios, desktop draws a thin orange interpolation line/triangle at the proportional position between them (mobile [mobile-only] simplifies this to highlighting the nearer line). Year axis → `config.y1_label`/`y2_label`. Cells use `computeSensitivityCell` + `formatSensitivityCell`; derived modes show "—" while `quotesLoading`. Empty state when `selectedTables` is empty for the company. Matrix headers use the same `#0a0a0a`/white treatment as the comps table header.

## Scenario-grid sensitivity — multi-axis Brent mesh (1..3 axes, multilinear)

> **Multi-axis redesign (2026-06-18) of the 1-D Brent grid.** The 1-D mesh had a single ambiguous axis (`x_value` = "Brent of which year?"). It is now a REGULAR mesh over **1..3 driver axes** (e.g. Avg Brent 2026 / 2027 / 2028+): the analyst runs their model over the FULL Cartesian product of per-axis levels and uploads, PER COMPANY, the target price at every mesh node; the frontend interpolates that mesh **MULTILINEARLY** (a 2^d corner blend — linear / bilinear / trilinear) as the analyst drags **one slider per axis**. Clean break — no dual-path 1-D code survives in the frontend; the DB migration `ALTER`ed the table and converted any legacy shape.

Source of truth: `supabase/migrations/20260612000000_stock_guide_scenario_grid.sql` (original table + RPC) + `supabase/migrations/20260618200000_stock_guide_scenario_grid_multi_axis.sql` (the multi-axis generalization: `y_value`/`z_value` columns, 5-col PK, 5-col RPC, legacy cleanup; owner `worker_supabase`) + the frontend below. A sensitivity table is a **SCENARIO GRID** when its `definition` carries a **`grid`** block; the dashboard then renders a multi-slider panel that interpolates the **target price + upside** live as the analyst drags, instead of the static matrix. Static tables (no `grid`) are unchanged.

### Data model — the mesh table + the grid block

- **`stock_guide_scenario_grid(sensitivity_id bigint, ticker text, metric text, x_value numeric, y_value numeric, z_value numeric, primary_value numeric)`** — PK `(sensitivity_id, ticker, metric, x_value, y_value, z_value)` (6-col since `20260619000000`); `metric NOT NULL DEFAULT 'target_price'` (matches a `definition.grid.outputs[].key`); FK → `stock_guide_sensitivities(id)` `ON DELETE CASCADE`. `x/y/z_value` are the driver levels (one per `definition.grid.axes` entry, in order; an UNUSED axis is always `0` — `NOT NULL DEFAULT 0`); `primary_value` = the metric's value at that mesh node (BRL/share for `target_price`, BRL mn for a flow / net income, …). **The SENSITIVE per-company points live here, NOT in the jsonb.** RLS enabled, **no policies** (same posture as every `stock_guide_*` table) — reads via the hide-aware RPC, writes via the service-role Excel-upload pipeline (replace-total snapshot per `sensitivity_id`, bypasses RLS). The 6-col PK btree covers the canonical read (filter by `sensitivity_id`, per-ticker-per-metric series ordered by the coordinate axes).

This block names **no company → it is NOT sensitive and carries no hide-strip**. **Axis binding:** an axis resolves its live "today" value via `driver_id` (registry lookup → `resolveDriverValue` → static `current_value` or dynamic live market value) OR, legacy, via `driver_key` (direct `marketValues[key]`). **Outputs:** `outputs[]` replaces the legacy single `output:"…"` string (which maps to `[{key:'target_price',mode:'upside',label:'Target price'}]`). Each output maps to a distinct `metric` in `stock_guide_scenario_grid` and interpolates on the SAME axis coordinates. **Positional column convention:** in the downloadable template Excel and the upload, the FIRST *d* columns are the axis coordinates **in axis (storage x/y/z) order**, then one column per company.

### RPC (`get_stock_guide_scenario_grid(p_sensitivity_id bigint)`)

`RETURNS TABLE(ticker text, metric text, x_value numeric, y_value numeric, z_value numeric, primary_value numeric)` (6-col since `20260619000000`), ordered by `ticker`, `metric`, then the coordinate axes, **hide-aware** (a non-admin only receives points for tickers flagged `is_visible`; admins get everything — the criterion mirrors `get_stock_guide_sensitivity_tables`). `SECURITY DEFINER SET search_path = public, pg_temp`, GRANT `anon, authenticated`. Wrapper `rpcGetStockGuideScenarioGrid(supabase, sensitivityId)` in `src/lib/rpc.ts` coerces all four numerics → `number`, coerces a blank/missing `metric` to `'target_price'`, and drops any row with a non-finite coordinate/value so the interpolator never sees `NaN`. `mapGridBlock` validates `axes` = an array of 1..3 `{driver_id?, driver_key?, label, unit, tmin?, tmax?, tstep?}` (an axis is valid when it has a positive `driver_id` OR non-empty `driver_key`; `label`/`unit` coerced to string; `tmin/tmax/tstep` coerced to number; duplicate binding keeps the first; >3 dropped; **0 valid → `null` → static matrix**) and `outputs` = an array of 1..6 `{key, mode, label}` (mode ∈ the value-mode enum, default `upside`; dedup by key; **legacy `output:"target_price"` with no `outputs` → `[{key:'target_price', mode:'upside', label:'Target price'}]`**).

### Interpolation (browser, pure — `src/lib/stockGuideSensitivity.ts`)

Two pure helpers (unit-tested in `src/lib/__tests__/stockGuideSensitivity.test.ts`):

- **`buildGridMesh(points: MeshPoint[], dim): GridMesh | null`** — collects the DISTINCT ascending levels per axis (exact equality — both the stored coords and the live slider value flow from the same numeric column; the upload script rounds to 6 decimals), maps each point's coords → a tuple of per-axis indices, last-write-wins on a duplicate tuple, returns `null` for an empty cloud. `dim` is the table's axis count, so a degenerate axis (a single level) is still a real axis. `GridMesh = { dim, levels: number[][], values: Map<indexTupleKey, number> }` (key = `"i"` | `"i,j"` | `"i,j,k"`).
- **`interpolateMesh(mesh, at: number[]): number | null`** — per axis: clamps a non-finite query to the axis min, brackets via binary search; a single level / `≤ first` → collapse at the lower edge (`frac 0`), `≥ last` → collapse at the upper edge, else `frac = (v−lo)/(hi−lo)` (span 0 → 0). It **COLLAPSES axes with `lo===hi` or `frac===0`** (it does NOT enumerate that axis's upper corner), so a border clamp on a complete mesh never demands a corner outside the active cell. It enumerates the `2^k` corners of the active cell (k = non-collapsed axes, ≤ 3), `weight = Π (frac | 1−frac)`, and returns `Σ weight · value`. **A required corner missing from the active cell → `null` (never `NaN`).**

**Upside** = `targetPrice / livePrice − 1` (guard `livePrice > 0`), reusing the dashboard's `liveByTicker` price index.

### Hook (`useStockGuideData.ts`)

The shared brain owns the grid state so **both Views drag the same sliders**:

- `isGridTable(table)` — true when `definition.grid != null`.
- **Lazy mesh fetch + cache:** when a selected table is a grid, the hook fetches `rpcGetStockGuideScenarioGrid(table.id)` once (cached by id, guarded by a fetched-set ref) and stores the raw `ScenarioGridPoint[]`. `gridLoading` is true while a selected grid's mesh is in flight.
- **`gridIndexById` is built ONCE per fetch in a `useMemo`** (`Record<tableId, Map<ticker, Map<metric, GridMesh>>>`, dimensioned by `axes.length`) — NEVER per drag. Each configured output interpolates its OWN `metric` mesh on the SAME axis coordinates.
- Per-table axis overrides (`gridAxisValuesById: Record<id, (number|null)[]>`): `null` = follow the live value. **LAZY** — never eager-initialised (market values arrive async), so a fresh axis stays `null` until dragged. `setGridAxisValue(tableId, axisIdx, v)`, `resetGridAxis(tableId, axisIdx)`, `resetGridAll(tableId)`.

### Excel upload (LONG format)

**Template the admin downloads (v2):** the browser generates **one sheet per output** (sheet name = the output `key`, e.g. `target_price`, `fcfe`, `dividends`, `net_income`). Within a sheet the layout is the canonical LONG one — **one row per scenario** = the FULL Cartesian product of the per-axis levels, the **FIRST *d* columns being the axis coordinates in axis (x/y/z) order** (positional convention; the header text is the axis label, NOT necessarily the driver key) + one column per ticker carrying that metric's value.

**Uploader contract (`scripts/manual/stock_guide_brent_grid_upload.py`, owner `worker_etl-pipelines`/`dados-locais`):** the script must (a) read **each sheet as a separate metric** (the sheet name = the `metric` written to `stock_guide_scenario_grid.metric`; default `'target_price'` for a single-sheet legacy book); (b) take the **first *d* columns positionally as the axis coordinates** in `grid.axes` order (do NOT rely on header text matching `driver_key` — the v2 axes may bind by `driver_id` and the header is the human label); (c) treat the remaining columns as per-ticker values (column header = ticker); (d) validate Cartesian completeness per sheet, round coords to 6 decimals, set unused coordinate columns to `0` when `d < 3`; (e) **replace-total-snapshot per `sensitivity_id`** (all metrics at once). **Payload guidance: keep ≤ 15 levels/axis in 3-D** (15³ ≈ 3.4k nodes/ticker/metric); ≤ 40×40 in 2-D; free in 1-D — multiply by the number of outputs.

### In-admin upload (browser — closes the configure → fill → upload loop)

Since 2026-06-19 the analyst no longer needs a terminal: the scenario-grid editor's **"Upload filled template (.xlsx)"** button (right beside "Download template", enabled only once the shell is saved) parses + validates the filled workbook **entirely in the browser** and uploads it via two `is_admin()`-guarded RPCs — same `stock_guide_scenario_grid` shape + replace-total semantics as the Python uploader, which stays as the automation fallback.

**DB contract (applied live, migration `20260619100000`):**
- `admin_replace_stock_guide_scenario_grid(p_sensitivity_id bigint, p_rows jsonb, p_first_chunk boolean) → integer` — `is_admin()`-guarded (`42501`). `p_first_chunk=true` does the replace-total DELETE of the id BEFORE inserting; every chunk ON CONFLICTs on the 6-col PK (idempotent retry). Validates in SQL: non-empty `ticker`/`metric`, 4 finite numerics, NaN rejected (`22023 nan_not_allowed`). `p_rows` shape (SHORT keys): `[{ticker, metric, x, y, z, v}]` (`v` = primary_value; `y`/`z` = 0 for an unused axis).
- `admin_count_stock_guide_scenario_grid(p_sensitivity_id bigint) → TABLE(total bigint, by_metric jsonb)` — post-upload confirmation.
- **NOT atomic across chunks** → the workbook is validated 100 % client-side BEFORE the first chunk; a mid-upload failure surfaces an error + a Retry that re-runs the whole replace (idempotent).

**rpc.ts wrappers (single writer, § Stock Guide):** `rpcAdminReplaceStockGuideScenarioGrid(supabase, sensitivityId, rows, firstChunk) → number` (returns rows written by the chunk) + `rpcAdminCountStockGuideScenarioGrid(supabase, sensitivityId) → { total, byMetric }` (coerces the `by_metric` jsonb to `Record<string, number>`). The upload-row type is `ScenarioGridUploadRow` (the SHORT keys).

**Browser parser/validator (`src/lib/stockGuideGridUpload.ts`, pure + unit-tested):** `parseScenarioGridWorkbook(workbook: ExcelJS.Workbook, grid: SensitivityGridBlock) → { rows, errors, warnings, summary }`. It is the TypeScript port of the Python uploader's validations, in lock-step:
- a sheet whose name (case-insensitive) matches an output `key` → that `metric`; an unknown sheet → **warning** (skipped); a configured output with no sheet → **warning** (absent);
- per sheet: the FIRST *d* columns are the axis coordinates read **positionally** (a header ≠ axis label → **warning**, not error); every coordinate cell must be numeric (**error** with up to 10 Excel row numbers), rounded to 6 decimals; a duplicate tuple → **error**; **Cartesian completeness** (`rows === Π distinct levels`; **error** with the count of missing combos);
- per ticker: a 100 %-empty column → **warning** (skipped), a partial column → **error**, a non-numeric cell → **error**;
- a 0-point total → **error** (refuse to wipe with nothing).
Returns the `rows` with the SHORT keys plus a `summary` (`{ byMetric, byTicker, totalRows, metricCount, tickerCount, scenarioCount }`). `chunkUploadRows(rows, 2000)` splits them into chunks (first → `firstChunk: true`). Unit tests: `src/lib/__tests__/stockGuideGridUpload.test.ts` (happy 2-sheet, broken completeness, partial ticker, 100 %-empty ticker, duplicate tuple, unknown sheet, 1-D y/z=0, non-numeric coord, empty book, `chunkUploadRows`).

**UI flow (admin-panel, desktop-only — `useAdminPanelData.ts` owns the state, `desktop/View.tsx` renders `GridUploadPanel`):** pick `.xlsx` → `parsing` (ExcelJS `workbook.xlsx.load(arrayBuffer)`) → **report** (errors in red BLOCK the Confirm, warnings in amber allow it, summary line "X scenarios × Y tickers × Z metrics = N points") → **Confirm upload (replaces all existing points)** → **uploading** in ~2000-row chunks (first = `firstChunk: true`) with progress ("Uploading 4,000 / 13,824…") → on finish the count RPC runs and shows "✓ N points uploaded (target_price: …, fcfe: …)" + bumps the read-only shell point count. A mid-upload failure shows a clear message + a Retry button (re-runs the whole replace). All copy is in English. Admin-panel has no mobile View → no dual-view obligation.
